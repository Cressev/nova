# Nova Runtime-first 产品形态重审

## 这份文档为什么存在

Nova 最初不是从“我要做一个开发 Agent”开始的。它一开始更像一个求职方向的想法：如果能抓取招聘信息、理解岗位要求、帮用户组织材料，也许能成为一个有用的企业级 Agent 应用。

但这个方向很快暴露出一个问题。招聘信息这件事，本身强依赖外部网站、登录状态、反爬策略和数据稳定性。如果抓不到、抓不全、抓不稳，产品价值就会突然塌掉。于是项目转向了一个更可控、更能体现工程能力、也更贴近用户长期需求的方向：做一个个人本地开发 Agent。

这个转向是对的。因为用户真正想要的不是一个只能聊天的工具，而是一个能在本地项目里工作、能读文件、能调用命令、能解释自己做了什么、能被长期信任的开发伙伴。

但是过去几轮开发也说明了一件事：Nova 不能只靠界面修修补补走到 Codex 或 Claude Code 的体感。现在的 Web 工作台已经有了项目切换、对话流、工具卡片、设置页、状态线和历史记录，但这些只是用户看见的表层。真正决定体验的，是背后有没有一个稳定的运行时。

运行时，通俗说，就是 Agent 真正“干活”的脑子和手脚：它要记住当前会话，决定什么时候问模型，什么时候调用工具，什么时候等待用户审批，什么时候取消任务，什么时候把工具结果整理给模型继续思考。

这份文档的目的，就是把 Nova 的下一阶段重新收束到这个核心上：做一个 Web-first 的个人本地 Coding Agent Runtime，同时把开源展示和自用体验都做到足够强。

## 我们已经走过的路

Nova 的第一段路，是把项目从一个想法变成可以打开的网站。这个阶段做出了 FastAPI 网关、静态 Web UI、GLM-4.7 模型接入、流式输出和基础会话。

FastAPI 是 Python 的 Web 框架，负责把浏览器请求接进后端。GLM-4.7 是当前使用的大模型。流式输出，就是模型边生成边显示，不用等完整回答结束。

这一段的价值是跑通了入口。用户可以打开 `http://127.0.0.1:8765`，在网页里和 Nova 对话。

第二段路，是把普通聊天页改成开发工作台。我们加了左侧项目和历史线程，中间对话流，右侧工作区、工具、记忆、配置和状态。这个阶段解决了很多“看起来不像开发 Agent”的问题。

但是问题也开始出现。用户每次指出的缺口，表面上是 UI 细节，比如工具折叠、会话折叠、路径选择、状态线、设置页、历史跨项目切换；本质上都是同一个问题：Nova 的前端已经想表现成一个成熟 Agent，但后端还没有一个足够成熟的 Agent 运行模型。

第三段路，是尝试向 Codex、OpenClaw、Hermes 和 cc源码靠拢。我们做了工具循环，也就是“模型提出工具调用，后端执行工具，再把结果交给模型继续回答”。我们还做了工具事件持久化，让刷新后还能看到工具过程。

这让 Nova 不再只是套壳聊天页。但当前的实现仍然偏薄。`CodexLikeAgentRuntime` 还是一个大类，里面同时负责模型决策、工具解析、工具执行、最终回答和一些特殊意图处理。它能跑，但还不像真正的 Agent 底座。

大类，通俗说，就是一个文件或类承担了太多职责。短期写起来快，长期会让功能越来越难加，异常越来越难兜，用户体验也越来越不稳定。

## 当前真正的问题

Nova 现在最危险的误区，是继续把精力放在“再加一个面板、再修一个按钮、再做一个折叠效果”上。

这些事情不是不重要。用户体验必须好，而且 Nova 既然要自己长期使用，体验就不能粗糙。问题在于，如果底层运行时没有补齐，再漂亮的界面也只能展示有限的信息。它不能稳定处理复杂任务，不能优雅中断，不能把多个工具调用排好队，不能在权限请求时暂停等待，不能在上下文太长时自动整理，也不能让用户真正信任它。

上下文，是模型当前能看到的内容，包括用户对话、项目记忆、工具结果和系统指令。上下文太长时，模型会变慢、变贵，甚至超出限制。成熟 Agent 会有压缩机制，也就是把旧内容整理成摘要，保留关键事实，释放空间。

这也是为什么 cc源码里的 `QueryEngine`、`StreamingToolExecutor` 和 `ToolUseContext` 值得参考。

`QueryEngine` 可以理解成“一个会话的总管”。它不只是处理一次请求，而是持续维护这个会话的消息、工具状态、权限、记忆和 token 用量。

`StreamingToolExecutor` 可以理解成“工具调度器”。它决定哪些工具可以一起跑，哪些必须单独跑；某个工具失败时，是否要取消旁边正在跑的工具；工具结果要按什么顺序交回给模型和前端。

`ToolUseContext` 可以理解成“工具执行时随身携带的上下文包”。工具需要知道当前项目在哪里、权限是什么、是否被用户取消、应该把进度发给谁、能不能访问 MCP 或记忆系统。

这些概念听起来偏工程，但它们直接决定用户体验。用户看到的是“工具是否流畅展示、能否取消、能否审批、是否乱跑命令、刷新后是否还能看见过程”。背后靠的就是这些运行时结构。

## 新定位

Nova 下一阶段的产品定位应该是：

Nova 是一个开源的 Web-first Personal Coding Agent Runtime。

这句话拆开看。

Web-first，意思是用户主要在网页里使用，而不是必须打开命令行。网页能提供更好的可视化，比如工具过程、权限审批、状态线、历史线程、工作区切换和 trace 回放。

Personal，意思是它优先服务个人开发者，不急着做多人企业平台。它要理解用户自己的项目、自己的偏好、自己的工作目录和自己的长期记忆。

Coding Agent，意思是它不是聊天助手，而是会在项目里完成开发任务的 Agent。它要能读代码、搜代码、看 diff、执行受控命令、写文件、解释改动、运行测试。

Runtime，意思是重点不是“做一个好看的聊天页面”，而是做一个能支撑 Agent 长期稳定工作的运行底座。

这个定位同时满足两个目标。

第一个目标是开源求职作品。面试官或开源读者能看到，Nova 不只是调了一个模型 API，而是在认真解决 Agent 工程里的核心问题：会话、工具、权限、记忆、trace、工作区和用户体验。

第二个目标是自己真的愿意用。如果 Nova 用起来不顺，开源包装再好也没有意义。体验要极致，意味着任务过程要清楚，工具状态要可信，错误要能解释，设置要好改，路径选择要自然，权限请求不能打断思路但也不能偷偷越界。

## 下一阶段的产品原则

Nova 后续开发要遵守一个顺序：先让 Agent 真的可靠，再让界面把可靠感表达出来。

这不是说 UI 不重要。恰恰相反，Nova 的优势就是 Web-first。只是 UI 要服务于运行时，而不是替运行时遮丑。

当后端有了真正的会话引擎，前端就可以展示“当前会话正在第几轮思考、哪些工具排队、哪个工具正在执行、哪个请求在等审批、上下文还剩多少、这次任务改了哪些文件”。这些展示会自然产生高级感，不需要靠装饰性的卡片堆叠。

当后端有了统一工具执行器，用户体验会变得稳定。多个只读工具可以并行，写入工具和 shell 命令会排队，失败时能清楚解释，取消时不会留下半截状态。

当后端有了权限模型，用户不会再遇到“该拦的不拦，不该拦的硬拦”的体验。权限模型，就是把工具能做什么、什么时候需要确认、确认是一次有效还是长期有效这些规则说清楚。

当后端有了 trace，用户和开发者都能复盘。trace 是执行轨迹，记录 Agent 在一个任务里经历了什么：模型说了什么、工具跑了什么、结果是什么、哪里失败、为什么停下。

## v1.0 重新定义

Nova v1.0 不应该再定义成“Codex-style 三栏工作台”。这个说法太偏 UI，也容易让开发走向复刻页面。

Nova v1.0 应该定义成“可验证的个人本地 Coding Agent Runtime，带 Web 工作台”。

可验证，意思是每个核心能力都要有测试、截图或 demo 脚本证明。比如工具并行不能只在文档里说支持，要有测试证明两个只读工具确实并行，写工具确实不会并行。权限审批不能只写一个配置项，要能在页面上暂停、确认、拒绝，并把结果写入会话。

个人本地，意思是默认围绕本机项目目录工作。Nova 要尊重本地文件安全，不越界访问，不把 API key 写进仓库，不偷偷 push，不随便删除文件。

带 Web 工作台，意思是网页仍然是主入口，但它的角色是“把运行时状态翻译成人能理解的界面”。它不是独立于后端的漂亮外壳。

## v1.0 的核心建设顺序

下一阶段第一件事是拆出 `AgentSession`。

`AgentSession` 可以理解成“一个对话线程的后台对象”。它保存这个线程的消息、工具事件、当前工作区、权限状态、上下文统计和取消信号。用户在网页上看到一个线程，后端就应该有一个对应的 session。

第二件事是实现 `ToolExecutor`。

`ToolExecutor` 是工具执行器。它不关心用户界面长什么样，只负责把工具调用排队、执行、取消、记录结果。它要知道哪些工具可以并行，哪些工具必须独占。比如读文件和搜索代码通常可以并行，写文件和执行 shell 命令就应该更谨慎。

第三件事是升级权限审批。

审批不是简单弹窗。它要支持“本次允许”“总是允许”“拒绝”“记住拒绝”。它还要能解释为什么需要审批，比如“这个命令会执行 shell”“这个操作会写入工作区外的路径”。

第四件事是做上下文预算和压缩。

上下文预算，就是估算这次对话还能放多少内容。压缩，就是当内容太多时，把旧的工具结果和对话整理成摘要。这个能力不需要第一版做到完美，但必须有可见状态和保底策略。

第五件事是做 trace 回放。

回放不是给开发者看的日志墙，而是用户能理解的任务故事。比如一次任务可以回放成：用户提出需求，Nova 制定计划，读取了三个文件，修改了两个文件，运行测试失败一次，修复后通过。

## 极致自用体验是什么

极致体验不是大而全，而是每一步都不别扭。

用户发送消息后，自己的消息必须立刻出现。Nova 开始工作后，状态不能空白，要告诉用户正在理解、正在读文件、正在等工具、正在整理回答。

工具调用必须默认能看懂。用户不应该点开一堆 JSON 才知道发生了什么。工具卡片应该先用一句人话说明“读取 README.md”“搜索 agent_runtime.py 中的 stream”，需要时再展开参数和原始结果。

路径选择必须像 IDE。输入路径时应该有候选、能键盘选择、能创建目录、能清楚区分“可切换”和“可新建”。

权限请求必须不烦人。安全是必要的，但不能乱拦。比如读取本地信息和执行受控命令要区别对待。用户明确要求查本机 WiFi 信息时，Nova 应该走受控工具链路，而不是自作聪明硬拒绝。

错误必须有解释。模型失败、工具失败、权限拒绝、路径不存在、命令超时，都要能显示原因和下一步建议。

设置必须像产品，不像临时配置页。模型、Base URL、上下文窗口、权限模式、网络访问、工具轮次这些都应该能修改，并清楚说明哪些需要重启。

## 开源展示应该怎么讲

Nova 的 README 和演示不应该说“我复刻了 Codex”。这句话容易显得浅，也有品牌风险。

更好的讲法是：

Nova 是一个 Web-first personal coding agent runtime. 它借鉴 Codex、Claude Code、OpenClaw、Hermes 和 cc源码中的成熟设计，但用 Python 后端和 Web 工作台重新实现，重点展示本地可控、工具执行、权限审批、项目记忆和 trace 回放。

开源展示要讲清楚三个故事。

第一个故事是产品故事：为什么从招聘 Agent 转向本地开发 Agent，因为后者更可控、更能长期使用，也更能体现工程深度。

第二个故事是工程故事：Nova 如何从聊天页一步步变成有运行时、有工具、有权限、有记忆、有 trace 的 Agent。

第三个故事是体验故事：用户不是看日志，而是在网页里看到 Agent 如何理解任务、调用工具、等待审批、修改文件和验证结果。

## 成功标准

如果下一阶段做对了，Nova 应该能完成一个真实演示。

用户打开网页，选择一个项目，输入“帮我理解这个模块并修一个小 bug”。Nova 会先给出计划，然后读取相关文件，必要时搜索代码，提出修改，等待用户批准写入，执行修改，运行测试，把失败和修复过程展示出来，最后给出简洁总结。刷新页面后，整个过程仍然能展开查看。

这个演示比单纯展示漂亮 UI 更有说服力。它能证明 Nova 是一个真正的 Coding Agent，而不是模型聊天套壳。

## 近期不做什么

Nova 近期不做多人协作平台。那会引入账号、权限组、组织空间、远程执行等复杂问题，会稀释当前目标。

Nova 近期不把重点放在招聘信息抓取。那个方向依赖不稳定外部数据，容易让项目价值被数据源限制。

Nova 近期不追求完全复刻某个产品的 UI。我们要学习成熟产品的结构和体感，但最终要形成自己的工程叙事。

Nova 近期也不做模型训练。trace 可以先用于复盘、调试和改进 prompt/skills，不急着做强化学习或微调。

## 文档写作约束

后续所有面向用户或开源读者的 Nova 核心文档，都应该遵守这次确认的写法。

文档要先讲故事，再讲结构。不能一上来丢术语。

如果必须提到新函数、新类或新概念，要立刻解释。比如提到 `AgentSession` 时，要马上说明它是“一个对话线程的后台对象”。提到 `ToolExecutor` 时，要马上说明它是“工具执行器，负责排队、并行、取消和记录结果”。

文档要循序渐进。不要从“用户体验不好”突然跳到“实现上下文压缩”。中间必须解释：体验不好是因为状态不可见，状态不可见是因为运行时没有统一事件模型，事件模型又依赖会话和工具执行器。

文档可以使用少量小节，但避免大段罗列。真正重要的不是把功能列满，而是让读者理解 Nova 为什么要这样做。

## 结论

Nova 下一阶段应该坚定选择 Runtime-first。

它不是放弃用户体验，而是把用户体验建立在可靠运行时上。一个好用的 Web 工作台，必须能展示真实、稳定、可恢复的 Agent 过程。一个好的开源求职作品，也必须让人看见背后的工程判断。

Nova 要做的不是“包装一个开源项目”，而是把多个成熟 Agent 产品和源码里的关键思想消化掉，用 Python 后端和 Web-first 体验做成一个自己真正愿意每天使用的个人 Coding Agent。
