## 2. 背景

用户希望开发一款个人使用的编码 Agent 产品，参考 Codex、Claude Code、OpenClaw、Hermes、OpenClaw-RL 等产品和项目，但不做简单复刻。

前期讨论中曾考虑过“企业级 Agent 应用”和“帮助找工作”的方向，也讨论过招聘网站信息抓取的可行性。结论是：招聘网站爬虫不稳定，合规和维护成本高，项目亮点容易偏向反爬而不是 Agent 产品能力。因此项目方向收敛为：

> 面向个人开发者的本地优先编码 Agent 产品。

该产品的核心价值不是“替用户聊天”，而是帮助个人开发者在长期项目中更好地管理上下文、执行工具、沉淀记忆、复用工作流、追踪 Agent 行为，并逐步形成适合个人习惯的开发助手。

## 3. 产品定位

Nova 是一个本地优先的个人开发 Agent 工作台，面向长期软件开发过程中的代码阅读、代码修改、调试、文档、审查、自动化和项目记忆管理。

产品应具备以下定位：

- 不是通用聊天机器人。
- 不是招聘网站爬虫。
- 不是企业多人 AgentOps 平台的第一版。
- 不是单纯模型 API 包装器。
- 是面向个人开发者的本地 Agent Harness。

其中 “Harness” 指 Agent 运行所需的完整工程外壳，包括上下文、工具、权限、记忆、技能、执行环境、日志、回放和评估。

## 4. 产品目标

### 4.1 长期目标

构建一个能够长期陪伴个人开发者工作的本地开发 Agent，使其能够：

- 理解项目结构和项目历史。
- 安全地读取、修改、运行和审查代码。
- 在跨会话场景中保留关键上下文。
- 将重复工作流沉淀为 Skills。
- 记录工具调用 trace，用于复盘和改进。
- 从真实使用过程发现重复失败和重复流程。
- 在用户确认后改进项目规则、记忆或 Skills。

### 4.2 第一阶段目标

第一阶段不追求复杂多端和训练能力，而是先完成可用、可控、可恢复的本地开发 Agent MVP：

- 支持 Web 网关和网页交互。
- CLI 只作为启动、诊断和备用功能。
- 支持项目记忆持久化。
- 支持项目指令加载。
- 支持文件读写和 shell 工具。
- 支持工具权限审批。
- 支持任务 trace 和 session 日志。
- 支持用户侧可观测前端，展示任务进度、工具调用、审批、trace 摘要和执行结果。
- 支持 Langfuse 开发调试观测，但不把 Langfuse 作为最终用户界面。
- 支持基础 resume。
- 支持手动 Skills。

## 5. 目标用户

### 5.1 核心用户

个人开发者，尤其是：

- 长期维护多个项目的个人开发者。
- AI Agent 应用开发者。
- 后端、全栈、工具链开发者。
- 希望构建自己长期工作流的开发者。
- 希望减少上下文重复解释成本的开发者。

### 5.2 非核心用户

第一阶段不优先服务：

- 企业多人团队。
- 非技术用户。
- 纯办公自动化用户。
- 需要复杂权限组织架构的企业管理员。
- 只想使用网页聊天助手的用户。

## 6. 核心问题

### 6.1 上下文丢失

现有 Agent 工具在长时间项目工作中容易出现上下文丢失：

- 新会话不知道项目历史。
- 压缩上下文后丢失关键决策。
- 重复解释项目结构和偏好。
- 重要规则只存在聊天里，没有落盘。

### 6.2 工具执行风险

编码 Agent 需要执行 shell、修改文件、访问网络和调用外部系统，这些能力存在风险：

- 误删文件。
- 覆盖用户未提交修改。
- 执行危险命令。
- 读取或泄漏敏感文件。
- 对外发送项目内容或凭证。

### 6.3 工作流不可复用

很多开发任务具有重复性，例如：

- 代码审查。
- 调试失败测试。
- 阅读新项目。
- 生成发布说明。
- 检查安全风险。
- 同步项目记忆。

如果每次都靠自由聊天完成，效率不稳定，也难以形成个人长期能力。

### 6.4 Agent 行为不可追踪

Agent 完成任务后，如果缺少 trace，很难回答：

- 它读了哪些文件？
- 它运行了哪些命令？
- 它为什么做这个修改？
- 它哪里失败过？
- 它是否遵守了权限规则？
- 下次如何避免同类错误？

### 6.5 个人偏好难以沉淀

用户对代码风格、文档风格、风险偏好、常用工具、默认技术路线有个人习惯。若没有稳定记忆和规则文件，Agent 很难持续适应用户。

## 7. 设计原则

### 7.1 本地优先

项目文件、记忆、日志、配置和 trace 默认保存在本地，用户可以检查、备份、删除和迁移。

### 7.2 安全默认

默认不静默执行高风险动作。文件写入、shell 命令、网络访问、敏感路径和外部系统调用需要明确权限策略。

### 7.3 记忆落盘

关键状态不能只存在聊天中。项目记忆、当前任务、进度、日志、任务入口和 TODO 追踪必须写入 durable 文件。

### 7.4 Skills 优先

重复任务应沉淀为可复用 Skills，而不是每次重新写 prompt。

### 7.5 可追踪

工具调用、审批、结果、失败和最终输出要能被追踪，用于复盘、评估和改进。

### 7.6 用户确认

涉及长期记忆、规则、Skills 和高风险操作的变更，应优先经过用户确认。

## 9. 使用场景

### 9.1 阅读新项目

用户进入一个项目，要求 Agent：

- 扫描项目结构。
- 阅读 README、配置、关键源码。
- 总结模块边界。
- 找出启动方式和测试方式。
- 写入项目记忆。

### 9.2 修改功能

用户提出功能需求，Agent：

- 读取项目规则。
- 搜索相关代码。
- 制定计划。
- 修改文件。
- 运行测试或构建。
- 输出变更说明。
- 更新任务日志。

### 9.3 调试错误

用户提供报错，Agent：

- 读取错误日志。
- 定位相关代码。
- 复现问题。
- 尝试修复。
- 记录根因。
- 将稳定坑点写入 findings。

### 9.4 代码审查

用户要求审查当前改动，Agent：

- 读取 git diff。
- 关注高风险问题。
- 输出按严重程度排序的 findings。
- 不直接修改工作区，除非用户要求修复。

### 9.5 项目记忆维护

用户完成阶段性工作后，Agent：

- 更新 CURRENT。
- 更新 PROGRESS。
- 追加 log。
- 记录 TODO 完成状态。
- 必要时更新 AGENTS 规则。

### 9.6 Skill 沉淀

当某类任务反复出现，Agent：

- 从 trace 中识别重复流程。
- 生成 Skill 草稿。
- 请求用户确认。
- 保存为可复用 Skill。
- 后续按触发条件调用。

## 10. 功能需求

### 10.1 本地 CLI/TUI

#### 需求描述

CLI/TUI 不再作为第一阶段主交互入口。第一阶段主入口调整为 Web 网关和网页交互，CLI 仅保留启动、诊断和备用能力。

#### 功能点

- 支持 `nova serve` 启动本地 Web 网关。
- 支持 `nova doctor` 检查环境、配置和依赖。
- 支持 `nova config` 查看或修改基础配置。
- 后续可扩展交互式 CLI/TUI。

#### 验收标准

- 用户可以通过 CLI 启动 Web 网关。
- CLI 不承担复杂交互界面。
- 复杂交互由 Web 前端承载。

### 10.1A Web 网关和用户前端

#### 需求描述

Nova 的第一阶段主交互入口是本地 Web 网关。用户通过浏览器与 Agent 交互，而不是直接使用复杂 CLI。

#### 功能点

- 支持浏览器聊天界面。
- 支持任务进度时间线。
- 支持工具调用卡片。
- 支持 shell 命令审批弹窗。
- 支持文件 diff 预览。
- 支持任务结果摘要。
- 支持会话历史。
- 支持本地 trace 摘要展示。
- 支持记忆文件状态展示。

#### 验收标准

- 用户可以打开网页发起任务。
- 用户能看到 Agent 当前正在做什么。
- 用户能批准或拒绝高风险工具调用。
- 用户能查看工具调用摘要和执行结果。
- 用户不需要打开 Langfuse 才能理解 Agent 执行过程。

### 10.2 项目指令加载

#### 需求描述

Agent 启动时读取项目中的长期规则和上下文文件。

#### 功能点

- 读取 AGENTS.md。
- 读取 CLAUDE.md。
- 读取 CURRENT.md。
- 读取 PROGRESS.md。
- 读取 log.md 末尾。
- 读取 user-queries.md 末尾。
- 读取相关目录 README。

#### 验收标准

- Agent 在开始任务前能加载项目规则。
- Agent 能识别用户偏好和硬规则。
- Agent 不会忽略产品研发文档“只有用户要求才写”的规则。

### 10.3 用户任务入口

#### 需求描述

通过 user-queries.md 记录用户请求原文、接收时间和完成时间。

#### 功能点

- 新请求追加到 user-queries.md。
- 使用 `---` 分隔任务块。
- 原样保留用户请求。
- 立即写入 `[Recieve:<timestamp>]`。
- 完成后写入 `[Done:<timestamp>]`。

#### 验收标准

- 每次用户任务都能在 user-queries.md 找到原文。
- Recieve 和 Done 时间完整。
- 不合并多个用户请求。

### 10.4 TODOList 任务追踪

#### 需求描述

通过 TODOList.md 持久追踪每次任务的计划、执行状态、失败记录和结束时间。

#### 功能点

- 每次任务追加 begin block。
- 记录用户请求原文。
- 创建分层任务清单。
- 执行过程中更新状态。
- 失败任务记录原因。
- 完成后追加 end block。

#### 验收标准

- TODOList.md 不重写历史记录。
- 状态从 `[]` 更新为 `[x]` 或 `[o]`。
- 任务结束后有完整 end 时间。

### 10.5 项目记忆

#### 需求描述

通过根目录核心文件维护项目连续性。

#### 功能点

- CURRENT.md 记录当前任务、状态、下一步和阻塞。
- PROGRESS.md 记录项目阶段级进展。
- log.md 记录关键动作和验证。
- findings/ 保存稳定发现。
- reports/ 保存用户明确要求的报告。
- references/ 保存工作流模板。

#### 验收标准

- 新会话可以通过这些文件恢复项目上下文。
- 关键状态不会只留在聊天中。
- 日志简洁，不保存大段命令输出。

### 10.6 文件工具

#### 需求描述

Agent 需要具备读取和修改项目文件的能力。

#### 功能点

- 读取文件。
- 搜索文件。
- 查看目录结构。
- 修改文件。
- 生成 patch。
- 检查改动结果。

#### 权限要求

- 默认只允许在 workspace 内操作。
- 敏感文件需要确认或禁止访问。
- 不得覆盖用户未授权变更。

#### 验收标准

- Agent 可以定位并读取相关文件。
- 文件修改可审查。
- 修改后能说明改了什么。

### 10.7 Shell 工具

#### 需求描述

Agent 需要通过 shell 运行测试、构建、搜索、格式化和诊断命令。

#### 功能点

- 运行只读命令。
- 运行测试命令。
- 运行构建命令。
- 捕获命令输出。
- 对长输出进行摘要。

#### 权限要求

- 高风险命令需要用户确认。
- 删除、覆盖、网络、权限提升类命令默认高风险。
- 不静默执行 destructive 命令。

#### 验收标准

- Agent 能说明将执行什么命令。
- 命令输出可追踪。
- 失败命令能记录失败原因。

### 10.8 权限引擎

#### 需求描述

定义并执行工具权限策略，减少本地 Agent 的风险。

#### 功能点

- 支持 read-only。
- 支持 workspace-write。
- 支持 shell-ask。
- 支持 network-ask。
- 支持 full-access。
- 支持路径 deny list。
- 支持命令风险识别。

#### 验收标准

- 文件写入必须经过 workspace 边界检查。
- 危险 shell 命令需要确认。
- 敏感文件默认不读或需要确认。

### 10.9 Trace 记录

#### 需求描述

记录 Agent 运行过程，支持复盘、调试和后续评估。

#### 功能点

- 记录用户请求。
- 记录加载的上下文。
- 记录模型选择。
- 记录工具调用。
- 记录审批结果。
- 记录工具输出摘要。
- 记录最终结果。
- 本地保存 JSONL trace。
- 异步上报 Langfuse 供开发调试。
- Web 前端展示用户可理解的 trace 摘要。

#### 验收标准

- 可以追踪一次任务中 Agent 做了什么。
- trace 不保存敏感明文。
- trace 可用于后续 replay 或 eval。
- Langfuse 不直接作为用户界面。
- 用户侧 trace 展示由 Nova Web 前端提供。

### 10.10 Session 恢复

#### 需求描述

支持从历史会话恢复工作上下文。

#### 功能点

- 保存 session 日志。
- 保存 session 摘要。
- 支持选择历史 session。
- 支持基于 CURRENT 和 PROGRESS 恢复任务。

#### 验收标准

- 新会话能理解上次任务状态。
- 用户不需要重复解释全部背景。

### 10.11 Skills 系统

#### 需求描述

将重复工作流沉淀为可复用 Skills。

#### 功能点

- 支持 Skill 文件结构。
- 支持手动调用 Skill。
- 支持 Skill 描述和触发条件。
- 支持 Skill 版本管理。
- 后续支持从 trace 生成 Skill 草稿。

#### 验收标准

- 用户可以创建和调用一个 Skill。
- Agent 能按 Skill 说明执行固定流程。
- Skill 不会自动写入长期规则，除非用户确认。

### 10.12 多模型 Provider

#### 需求描述

支持不同模型服务，避免绑定单一 Provider。

#### 候选 Provider

- OpenAI。
- Anthropic。
- Kimi。
- OpenRouter。
- 本地 OpenAI-compatible endpoint。

#### 验收标准

- 可以通过配置选择 Provider。
- 工具调用协议有统一适配层。
- 模型切换不影响核心工作流。

### 10.13 产品研发文档管理

#### 需求描述

产品研发文档集用于存放需求、调研、版本文档和发布文档，但文档正文不能自动生成。

#### 功能点

- README 维护目录说明。
- 普通文档默认空白。
- 只有用户明确要求写某份文档时才写正文。
- 新增文档使用实际创建日期。
- 新增文档后更新 README 文件清单。

#### 验收标准

- Agent 不主动补写普通产研文档。
- 用户明确要求写需求文档时，只写需求文档。
- 其他产研文档保持空白。

## 11. MVP 范围

### 11.1 MVP 必做

- Web 前端。
- Python FastAPI 本地网关。
- CLI 启动和诊断命令。
- 项目指令加载。
- 根目录项目记忆读取。
- 用户任务入口记录。
- TODOList 任务追踪。
- 文件读取和搜索。
- 文件写入和 patch。
- Shell 工具。
- 权限审批。
- 基础 trace。
- 用户侧可观测界面。
- Langfuse 开发调试上报。
- session 日志。
- 手动 Skill 调用。

### 11.2 MVP 可延后

- 消息平台 gateway。
- 子 Agent 并行。
- MCP。
- 自动 Skill 生成。
- replay eval。
- Docker sandbox。
- 多用户权限系统。

### 11.3 MVP 不做

- 招聘网站爬虫。
- 企业级多人 AgentOps 平台。
- RL 训练。
- fine-tune。
- 自动执行高风险命令。
- 默认联网读取私有数据。

## 12. 非功能需求

### 12.1 安全性

- 默认不执行破坏性命令。
- 默认不读取敏感文件。
- 默认不外传项目文件。
- 所有高风险工具调用可审计。
- 用户可配置 deny list。

### 12.2 可恢复性

- 会话中断后可从文件记忆恢复。
- 任务完成状态可从 TODOList 查看。
- 当前任务状态可从 CURRENT 查看。

### 12.3 可维护性

- 模块边界清晰。
- 工具实现和权限策略分离。
- Provider 适配层独立。
- Skills 与普通项目记忆分离。

### 12.4 可观测性

- 工具调用可追踪。
- 命令失败可定位。
- 关键动作写入 log。
- 长输出不污染长期记忆。

### 12.5 本地可控

- 项目记忆可读写。
- trace 可检查。
- 配置可导出。
- 用户可删除或迁移本地状态。

## 14. 权限边界

### 14.1 文件权限

- workspace 内读取默认允许。
- workspace 内写入需要路径检查。
- workspace 外操作需要确认。
- `.env`、SSH key、token 文件默认敏感。

### 14.2 Shell 权限

高风险命令包括但不限于：

- 删除文件或目录。
- 覆盖配置。
- 修改权限。
- 安装依赖。
- 网络上传。
- 远程执行。
- git 历史重写。

### 14.3 网络权限

- 默认不主动联网。
- 查询当前资料时需要说明来源。
- 涉及外部系统登录、token、私有数据时必须确认。

## 17. 风险与约束

### 17.1 产品范围风险

本项目容易膨胀为完整 Codex/Claude Code/Hermes 复刻。需要坚持 MVP 优先，先做本地 TUI/CLI、记忆、权限、trace。

### 17.2 安全风险

本地 Agent 能力越强，误操作风险越高。权限引擎必须早于复杂自动化能力实现。

### 17.3 记忆污染风险

错误规则、错误总结、过时信息可能写入长期记忆。长期记忆更新需要谨慎，重要规则应用户确认。

### 17.4 多 Provider 风险

不同模型的工具调用、上下文、输出格式不一致，需要统一适配层，避免核心逻辑散落在 Provider 分支中。

### 17.5 UI 复杂度风险

TUI 做到 Codex/Claude Code 级别需要较高工程投入。MVP 应先保证工作流闭环，再提升交互体验。
