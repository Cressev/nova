# Nova v1.0 Codex App 风格功能开发文档

## 1. 文档目标

本文件用于指导 Nova v1.0 从“普通聊天页”升级为“个人开发 Agent 工作台”。参考对象是 OpenAI Codex App 的公开官方说明，但 Nova 不复制 OpenAI/Codex 的品牌、商标、专有图标、原始文案或精确视觉外观。

本阶段目标是复刻 Codex App 的核心产品理念和交互结构：

- 项目与线程并行管理。
- 对话、计划、执行状态在同一工作台内可见。
- 本地、隔离、云端等运行模式在入口处明确表达。
- Git diff、任务摘要、终端、浏览器预览等开发上下文成为一等信息。
- 权限、沙箱、审批和验证结果持续可见。

## 3. Codex App 产品理念提炼

### 3.1 线程不是聊天记录，而是工作上下文

Codex App 把 thread 作为核心单元。一个 thread 绑定项目、目录、模式、上下文、执行轨迹和 Git 变化，而不只是问答消息。

Nova v1.0 对齐方式：

- 左侧展示项目、线程和最近活动。
- 中间展示当前 thread 的对话、计划和输出。
- 右侧展示当前 thread 的工作区状态、Git diff 摘要、审批和验证。

### 3.2 开发 Agent 需要工作台，而不是聊天框

Codex App 内置 Git、终端、浏览器预览、review pane、worktree 和 automation。它的核心是让用户在一个界面中看见“Agent 正在做什么、改了什么、下一步需要谁决策”。

Nova v1.0 对齐方式：

- 增加右侧任务侧栏。
- 增加 Git 状态卡片。
- 增加运行模式选择。
- 增加计划、审批、验证和输出摘要槽位。

### 3.3 模式选择必须前置

Codex App thread 可以选择 Local、Worktree、Cloud。模式决定 Agent 是否直接改当前目录、是否隔离变更、是否远程运行。

Nova v1.0 对齐方式：

- v1.0 先实现模式 UI：本地、工作树、云端。
- 当前真实执行只支持“本地”。
- 工作树和云端先展示为即将支持，避免误导用户。

### 3.4 权限和沙箱是产品体验，不是后端细节

Codex 的审批和沙箱会影响用户信任。用户需要知道 Agent 当前能读写哪里、是否能联网、是否需要审批。

Nova v1.0 对齐方式：

- 右侧展示权限摘要：工作区写入、网络访问、审批策略。
- 未来工具执行前展示审批卡片。
- 高风险操作必须停下来让用户确认。

### 3.5 Git review 是开发闭环

Codex App 的 review pane 不是附属功能，而是用户验收 Agent 工作的入口。它支持看 diff、评论、stage、revert、PR 反馈等工作流。

Nova v1.0 对齐方式：

- v1.0 先实现 Git 状态摘要：分支、未提交文件数、变更文件列表。
- 后续再做文件级 diff、hunk 操作和 inline comment。

### 3.6 浏览器预览让前端反馈具体化

Codex App 的 in-app browser 用于预览本地服务、截图、浏览器评论和让 Agent 操作页面。

Nova v1.0 对齐方式：

- v1.0 先放置浏览器预览槽位和本地 URL 输入概念。
- 本轮实现不内嵌真实浏览器控制，以 Playwright 验证当前页面。
- 后续接入 Browser 工具、截图和元素评论。

## 4. v1.0 功能范围

### 4.1 本轮必须实现

- 三栏工作台布局：
  - 左栏：品牌、项目、运行模式、线程列表、技能入口。
  - 中栏：线程标题、状态、消息流、计划提示、Composer。
  - 右栏：工作区状态、Git 变更摘要、运行与验证、审批策略。
- Composer 功能：
  - 输入消息。
  - 发送后立即显示用户消息。
  - 模型回复流式输出。
  - 支持 `/plan`、`/review`、`/status`、`$skill` 等快捷提示。
- 工作区状态接口：
  - 返回项目根路径。
  - 返回 Git 分支。
  - 返回未提交变更文件。
  - 返回运行模式能力。
- 前端状态展示：
  - 模型是否配置。
  - 网关是否在线。
  - 当前模式。
  - Git 变更数量。
  - 验证命令建议。
- 视觉要求：
  - 保持 Nova 原创品牌。
  - 参考 Codex 的工作台信息密度。
  - 不复制 Codex 专有外观、商标和文案。

### 4.2 v1.0 暂不实现

- 真正的 Git worktree 创建。
- 真正的云端运行环境。
- 内嵌浏览器控制。
- 终端多标签。
- inline diff 评论。
- stage、unstage、revert 操作。
- Appshots。
- 自动审批 reviewer。
- MCP 设置页面。

### 4.3 后续版本扩展

- v1.1：工具执行、审批卡片、真实文件读写、shell 命令。
- v1.2：Git diff pane、hunk 级操作、inline comment。
- v1.3：内嵌浏览器预览、截图、页面批注。
- v1.4：worktree 隔离线程。
- v1.5：Skills、MCP、自动化和长期目标模式。

## 5. 信息架构

### 5.1 左栏

- Nova 品牌。
- 当前项目卡片。
- 模式切换：本地、工作树、云端。
- 新线程按钮。
- 线程搜索入口占位。
- 最近线程列表。
- Skills 入口占位。

### 5.2 中栏

- 当前 thread 标题。
- 运行状态和模型状态。
- 计划提示条。
- 消息流。
- Composer。
- Slash command 提示。

### 5.3 右栏

- Workspace：
  - 项目路径。
  - Git 分支。
  - 变更文件数。
- Review：
  - 变更文件列表。
  - 后续进入 diff pane。
- Run：
  - 测试命令。
  - 启动命令。
  - 最近验证结果。
- Permissions：
  - 工作区写入。
  - 网络访问。
  - 审批策略。
- Browser：
  - 本地预览 URL。
  - 后续浏览器评论入口。

## 6. 接口设计

### 6.1 `GET /api/workspace/status`

返回当前工作区摘要。

```json
{
  "project_root": "/path/to/project",
  "git": {
    "available": true,
    "branch": "main",
    "dirty_count": 3,
    "files": [
      { "path": "static/app.js", "status": "M" }
    ]
  },
  "modes": [
    { "id": "local", "label": "本地", "enabled": true },
    { "id": "worktree", "label": "工作树", "enabled": false },
    { "id": "cloud", "label": "云端", "enabled": false }
  ],
  "permissions": {
    "workspace_write": true,
    "network_access": false,
    "approval_policy": "按需审批"
  },
  "commands": {
    "test": "PYTHONPATH=src python3 -m unittest discover -s tests",
    "serve": "PYTHONPATH=src python3 -m nova_gateway.cli serve --host 127.0.0.1 --port 8765"
  }
}
```

## 7. 验收标准

- 打开 `http://127.0.0.1:8765` 后，首屏是三栏开发 Agent 工作台。
- 用户能继续发送消息并看到流式输出。
- 用户消息必须立即显示。
- 右侧能看到当前项目、Git 分支和变更数量。
- 服务启动后 `/api/provider` 显示 `configured=true`。
- `/api/workspace/status` 能返回项目状态。
- 单元测试通过。
- Playwright 能打开页面、发送消息、截图、确认控制台无错误。

## 8. 风险与约束

- “1:1 复刻”只能用于功能结构和交互模式，不能复制 Codex 专有视觉、品牌、商标或原始文案。
- v1.0 当前后端仍是学习阶段，复杂工具执行和审批闭环要分阶段实现。
- Git 状态读取只能做只读摘要，不能在没有明确用户操作前执行 stage、revert、push 等有副作用动作。
- API key 仍只能通过环境变量注入，不得写入文档、代码、日志或 Git。
